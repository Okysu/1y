# 事务内存 (STM)

Actor 模型用隔离消灭了共享，但有些场景确实需要**共享**——多个组件读写同一份配置、跨多个数据结构做原子更新、生产者与消费者共享一个缓冲区。1y 为这类场景提供**软件事务内存（Software Transactional Memory，STM）**：用"事务"这个程序员早已熟悉的概念，取代难以推理的"锁"。

本页介绍 STM 的全部语法：`shared` 创建事务性单元、直接读写访问、`transact` 开启事务、`retry` 重试，以及嵌套事务与事务外的访问规则。

## 创建事务性单元：shared

`shared expr` 创建一个**事务性单元（transactional cell）**，它的值可以在事务中被读写。`shared` 返回一个 cell 句柄，而不是值本身。

```1y
let counter = shared 0;
let name = shared "anon";
let config = shared { host: "localhost", port: 8080 };
```

`shared` 接受任意 1y 值作为初始值。单元本身是**共享的引用**——多个事务可以同时读写它，但所有访问都受 STM 的隔离与原子性保护。你可以把一个 cell 想象成"一个可替换的标签"：标签指向的值不可变，但标签本身可以被换成指向新值。

## 读写单元

对 cell 的读写**与普通变量语法相同**——不需要任何前缀运算符。绑定名就指向 cell；读取它得到当前值，赋值给它则替换值：

- `counter` 读取单元的当前值。
- `counter = expr` 把单元的值设为 `expr` 的结果。

```1y
let v = counter;       # 读
counter = v + 1;       # 写
```

语义取决于上下文：在事务内，读写走快照隔离；在事务外，读写直接生效（见[事务外访问](#事务外访问)）。这种统一的语法让你可以在事务内外复用同一段读写逻辑，只需决定是否把它包进 `transact`。

## 事务：transact

`transact { ... }` 开启一个事务，把其中对 cell 的所有读写纳入**快照隔离**与**原子提交**。

```1y
let counter = shared 0;
let result = transact {
    let v = counter + 1;
    counter = v;
    v           # 事务的返回值
};
```

事务的关键性质：

- **快照隔离**：事务开始时对它将读到的 cell 拍下快照，整个事务期间读到的都是这份一致快照，不会被其他事务的中间状态污染。
- **原子提交**：事务结束时，若它读过的 cell 都未被其他事务改动，则所有写入一次性提交；否则整体回滚并重试。
- **有返回值**：`transact` 块最后一个表达式的值作为事务结果返回。提交成功后，调用方拿到这个值。

## 快照隔离

快照隔离意味着事务**看到的是它开始时刻的一致状态**，即便其他事务在此期间提交了改动。这使得事务内的逻辑可以写得像顺序代码一样直观：

```1y
# 原子转账：要么两个余额都更新，要么都不更新
let alice = shared 100;
let bob = shared 50;

transact {
    let a = alice;
    let b = bob;
    if a >= 30 {
        alice = a - 30;
        bob = b + 30;
    }
};
```

无论有多少事务并发运行，转账永远不会出现"钱从 alice 扣了却没到 bob"的中间状态。每个事务要么看到转账前的快照，要么看到转账后的快照，绝不会看到半截状态。这与数据库的 `SERIALIZABLE`/`REPEATABLE READ` 隔离级别精神一致，只是作用对象是内存中的 cell 而非数据表。

## 提交与回滚

事务的提交协议是**乐观并发控制**：

1. 事务执行期间不加任何锁，只在私有快照上读写。
2. 提交时，STM 检查事务读过的每一个 cell——它们的当前值是否仍与快照一致。
3. 若全部一致，提交成功，事务的所有写入原子生效。
4. 若有任何一个 cell 已被别的事务改过，提交失败，事务**整体回滚**并从头重试。

这种"先做后查"的策略在冲突不频繁时效率很高：绝大多数事务一次提交成功，没有锁开销。只有在高冲突时才会反复重试。配合 1y 的不可变数据，回滚几乎没有成本——快照本身是不可变值，丢弃重试即可，无需"撤销"任何原地修改。

## 重试：retry

`retry` 显式地放弃当前事务并从头重跑。它通常用于"条件不满足，等会儿再试"的模式：事务发现某个前置条件暂时不成立，主动 `retry`，等到相关 cell 发生变化后再试一次。

```1y
# 等待账户余额充足后再扣款
transact {
    if alice >= amount {
        alice = alice - amount;
    } else {
        retry        # 余额不够，重试事务
    }
};
```

要点：

- `retry` 会丢弃本次事务的所有写入，回到 `transact` 块开头重新执行。
- 为防止死循环，事务的**最大重试次数为 64**。超过上限仍无法提交时，事务最终失败并报错。
- `retry` 与自动回滚不同：自动回滚由冲突触发，`retry` 由程序员主动触发，二者最终都走"重新执行"的同一条路径。

## 嵌套事务

1y 支持嵌套事务：在 `transact` 内部再开一个 `transact`。**内层事务提交时，其写入提交到外层事务的私有快照，而非全局**；只有最外层事务提交成功，所有写入才真正对外可见。

```1y
transact {
    counter = counter + 1;
    transact {
        # 内层提交到外层快照；若外层回滚，这里也一并回滚
        counter = counter + 10;
    };
    # 此处读到的是外层快照，已包含内层的写入
    counter
};
```

嵌套事务的语义：

- 内层事务的回滚只影响内层，外层不受影响（可以视作内层从未发生）。
- 内层事务的提交不立即对外生效，而是并入外层。外层回滚则内层的写入一并回滚。
- 这与数据库的 savepoint 语义一致：内层是外层的一个原子子单元。

嵌套事务让"组合"变得自然：你可以把一个复杂的事务拆成若干小事务，每个小事务自身原子，再由外层事务把它们粘成一个大原子单元，而无需手写补偿逻辑。

## 事务外访问

在 `transact` 块之外，读写 cell 仍然可用，但语义不同：

- **事务外读** `counter`：直接读取单元的当前值，无快照、无重试。
- **事务外写** `counter = expr`：直接写入，立即生效，不经过事务协议。

```1y
let counter = shared 0;
counter = counter + 1;   # 事务外：直接读改写，非原子
```

事务外的访问是"逃生舱"，适合单线程初始化、一次性赋值等确定无竞争的场景。**只要存在并发访问，就应放进 `transact`**，否则会绕过 STM 的隔离保证，引入数据竞争。

## 完整示例：并发安全的计数器

```1y
import io;

let counter = shared 0;

# 多个事务并发自增，STM 保证最终结果正确
fn bump() {
    transact {
        let v = counter + 1;
        counter = v;
        v
    }
};

println("自增后：" + str(bump()));
println("自增后：" + str(bump()));
```

即便 `bump` 被并发调用，每个事务的"读-改-写"也是原子的：冲突的事务自动重试，最终 `counter` 的值恰好等于调用次数，不会丢失更新。对比锁版本——那里你需要小心地选择锁粒度、避免死锁——STM 版本几乎没有需要你操心的并发细节。

## 小结

| 元素 | 语法 | 作用 |
|------|------|------|
| 创建单元 | `shared expr` | 创建事务性 cell |
| 读 | `counter` | 读当前值（事务内走快照） |
| 写 | `counter = expr` | 写值（事务内纳入提交） |
| 事务 | `transact { ... }` | 快照隔离 + 原子提交 |
| 重试 | `retry` | 主动重跑事务（上限 64 次） |
| 嵌套 | `transact` 内 `transact` | 内层提交到外层 |

STM 的价值在于**原子性是组合的**：你可以把两个独立的事务拼成一个更大的事务，而无需担心锁顺序。配合 1y 的不可变数据，事务的重试是幂等的，既正确又高效。当状态适合封装时用 [Actor](./actors)，当需要跨边界协调共享状态时用 STM——二者共同构成 1y 的并发模型。
