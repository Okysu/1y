# 并发模型

并发是软件工程中最难的领域之一,而它的难度往往被归咎于"并发本身"。1y 的立场不同:并发的难度,大部分来自**错误的抽象**——共享可变内存 + 锁。1y 用两种经过验证的抽象取代了它:**Actor 模型**负责隔离状态与消息传递,**软件事务内存(STM)**负责共享状态的协调访问。两者都建立在不可变数据之上,共同构成了一个简单而安全的并发模型。

## 为什么需要重新思考并发

先看看主流并发模型的痛点:

- **线程 + 锁**:数据竞争、死锁、活锁、锁粒度难调。即使是最优秀的程序员也难以写出正确的锁代码。锁不组合:两个各自正确的加锁模块组合在一起,可能产生死锁。
- **async/await**:解决了阻塞,却引入了"函数着色"——同步函数与异步函数是两个世界,不能自由互调。还带来了 `Pin`、`Future` 生态、执行器选择等额外复杂度。
- **CSP(Go 的 channel)**:比锁好得多,但仍基于共享运行时与 goroutine 调度,且 channel 本身是共享可变状态的一种形式,需要小心地避免 deadlock。

1y 的判断是:**隔离比同步更简单**。与其让多个执行流共享一块内存再小心翼翼地同步,不如让每个执行流拥有自己的内存,通过传递消息来通信。这正是 Actor 模型的核心思想。

## Actor 模型:隔离 + 消息

Actor 是 1y 并发的第一块基石。一个 Actor 是一个**拥有私有状态的轻量进程**。它:

- 拥有完全隔离的状态,外部无法直接访问;
- 拥有一个信箱(mailbox),按顺序接收消息;
- 一次只处理一条消息,因此自身状态天然无竞争。

```1y
actor Counter {
  state count: Int = 0

  on Incr() { count = count + 1 }
  on Decr() { count = count - 1 }
  on Get() { reply(count) }
}
```

创建并使用一个 Actor:

```1y
let counter = spawn Counter()
counter ! Incr()                    // 发后即忘(fire-and-forget)
counter ! Incr()
yield;                              // 清空待处理的 `!` 消息
let n = counter ? Get()             // 请求/回复,返回 2
```

注意两种运算符的语义:

- **`!`(发后即忘)**:把消息投入信箱,立即返回。发送方不等待结果。适用于"命令"型消息——告诉 Actor 做某事,但不关心何时完成。
- **`?`(请求/回复)**:发送消息并等待回复。底层是一条 `!` 加上一个承诺(promise),回复通过约定的通道返回。适用于"查询"型消息——需要 Actor 的状态或计算结果。

因为每个 Actor 一次只处理一条消息,它的内部状态永远不存在并发访问。**你写 Actor 内部代码时,就像在写单线程程序**——这是 Actor 模型最大的心智简化。

### Actor 天然组合

Actor 之间可以互相发消息,形成任意的拓扑。一个 Actor 可以 spawn 子 Actor,把任务分派出去,收集结果。由于状态隔离,这种组合不会产生锁组合时的死锁问题:

```1y
// 一个池子管理者,把任务分派给 worker
actor Pool {
  state workers = []

  on Submit(task) {
    let w = pick_idle(workers)
    w ! task
  }
}
```

## STM:共享状态的协调访问

有些场景下,纯消息传递不够顺手——例如多个组件需要读写同一份共享配置,或需要跨多个数据结构进行原子更新。1y 为这类场景提供**软件事务内存(STM)**。

STM 让你把一段代码标记为"事务",这段代码对共享单元(用 `shared` 创建)的读写会获得**快照隔离**:事务看到的是事务开始时的一致性快照,事务提交时若发现单元已被其他事务改变,则自动重试。

```1y
// 两个账户,要原子地转账
shared alice = 100
shared bob   = 50

transact {
  alice = alice - 30;
  bob   = bob   + 30
}
// 要么两个更新都成功,要么都回滚——永远不会出现钱凭空消失
```

STM 的核心价值在于**原子性是组合的**:你可以把两个独立的 STM 操作拼成一个更大的事务,而无需担心它们之间的锁顺序。对比锁,这是质的飞跃——锁不组合,STM 组合。

```1y
// 组合多个 STM 操作成一个事务
transact {
  let balance = alice;
  if balance >= amount {
    alice = balance - amount;
    ledger = push(ledger, Transfer(alice, bob, amount));
    bob = bob + amount
  }
}
```

## Actor 与 STM 的分工

Actor 和 STM 并非二选一,而是分工互补:

| 场景 | 选用 | 理由 |
|------|------|------|
| 长生命周期的有状态服务 | Actor | 状态封装在 Actor 内,自然模块化 |
| 请求/响应式交互 | Actor(`?`) | 一一对应的语义清晰 |
| 跨多组件的原子更新 | STM | 事务组合性好 |
| 共享只读配置 + 偶尔更新 | STM | 避免为配置单独建 Actor |
| 大量独立 worker | Actor | 隔离更彻底,故障可监督 |

经验法则:**默认用 Actor,只有当你需要跨边界原子地协调多个共享引用时才用 STM**。这样能让系统的状态边界清晰可循。

## 快照隔离为什么安全

STM 的安全性来自两个性质:

1. **不可变数据**:1y 的值不可变,`shared` 单元只是"可替换的标签",替换时旧值仍然完整存在。事务读取的是某个时刻的快照,不会被半修改。
2. **乐观并发 + 重试**:事务执行时不加锁,提交时检查它读过的单元是否仍然持有原来的值。若被改过,事务整体重试。只要冲突不频繁,这比加锁更高效。

由于不可变性,重试是**幂等**的——重跑事务没有副作用(事务内用 `!` 发送的消息会在提交成功后才真正投递)。这是 1y 把不可变性与 STM 结合的精妙之处:不可变性让 STM 的实现既正确又简单。

## 与其他模型的对比

| 维度 | 线程+锁 | async/await | CSP | 1y(Actor+STM) |
|------|---------|-------------|-----|----------------|
| 数据竞争 | 常见 | 仍可能 | 较少 | 不可能(隔离) |
| 死锁 | 常见 | 少见 | 可能 | 极少(无锁) |
| 函数着色 | 无 | 严重 | 无 | 无 |
| 组合性 | 差 | 中 | 中 | 好(STM) |
| 心智负担 | 高 | 高 | 中 | 低 |

## 为什么 Actor + STM 更简单、更安全

回到开头的判断:**隔离比同步简单**。Actor 把"共享"这件事从根上消灭了——没有共享,就没有竞争。当确实需要共享时,STM 用"事务"这个程序员早已熟悉的概念(来自数据库)取代了"锁"这个难以推理的概念。

最终结果是,1y 的并发代码读起来像顺序代码:Actor 内部是顺序的,STM 事务内部是顺序的。并发只体现在**拓扑**——谁给谁发消息、谁 spawn 了谁——这个层面的设计本身就比"哪段代码加哪把锁"清晰得多。这就是 1y 对并发的承诺:**让正确成为默认,而非奇迹**。
