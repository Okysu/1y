# 数值统一

在绝大多数语言里,"数字"是一个支离破碎的概念。`int`、`long`、`float`、`double`、`BigInteger`、`BigDecimal`……程序员被迫在五六种数值类型之间做选择,而选错的代价往往是灾难性的。1y 用一个简单的决定终结了这种碎片化:**只有两种数值类型,且都是任意精度**。

## 两种类型,一种算术

1y 提供两种数值类型:

- **`Int`** —— 任意精度整数。没有上限,没有下限。
- **`Decimal`** —— 任意精度十进制小数。没有精度损失。

它们之间的关系由一条简单的规则 governing(支配):**运算结果总是"更宽"的那个类型**。`Int` 与 `Decimal` 运算时,`Int` 自动提升为 `Decimal`。这条规则让程序员从类型转换的泥潭中解放出来。

```1y
let a = 10                  # Int
let b = 3.14                # Decimal
let c = a + b               # Decimal,13.14
let d = a * b               # Decimal,31.4
```

## 除法的提升规则

除法是最容易引发 bug 的运算符,因为不同语言对"整数除以整数"的处理大相径庭。C、Java、Go 返回整数商(截断),Python 返回浮点,JavaScript 一切皆浮点。1y 采取了一条语义最诚实的规则:**能整除就返回 `Int`,不能整除就提升为 `Decimal`**。

```1y
let x = 10 / 2              # Int,5
let y = 10 / 3              # Decimal,3.3333...
let z = 7 / 2               # Decimal,3.5
```

这条规则的关键在于:**结果永远反映数学真相**。`10 / 3` 不是 `3`,不是 `3.3333333333333335`(浮点近似的尾巴),而是一个真正的十进制小数。你不会因为"忘记把操作数转成浮点"而丢失精度。

## 没有溢出,真的没有

在 1y 中,以下代码是完全合法且正确的:

```1y
let big = 10 :pow 100
# 这是一个 1 后面跟着 100 个 0 的整数
# 在 Java 中,这会让 long 溢出成一个奇怪的负数
# 在 JavaScript 中,这会变成 1e+100,丢失整数精度
# 在 1y 中,它就是它应当是的样子

let huge = factorial(500)
# 500 的阶乘是一个 1135 位的整数
# 1y 从容地持有它、运算它、打印它
```

对比一下其他语言中因溢出引发的真实事故:2014 年《糖果传奇》因金币计数器溢出导致玩家资产归零;2015 年波音 785 的电力控制因 `int16` 溢出在 248 天后宕机;无数智能合约因 `uint256` 下溢被攻击者掏空。这些事故的根源不是程序员粗心,而是**语言把硬件的字长暴露给了业务逻辑**。1y 拒绝这种暴露。

## 任意精度,但不必担心性能

一个常见的担忧是:任意精度是不是很慢?答案是——**对绝大多数业务代码,完全可以忽略不计**。1y 的 `Int` 在内部对小数值使用机器原生整数表示,只有在数值增长超过机器字长时才切换为大数表示。这意味着 `1 + 1` 的开销与原生整数几乎相同,而 `factorial(500)` 才会触发大数路径。

```1y
# 日常运算,走快速路径
let sum = List.range(0, 1000) |> List.sum   # Int,499500

# 巨数运算,走大数路径
let pow1000 = 2 :pow 1000                    # 302 位的整数
```

`Decimal` 同样如此:常用的小数被高效表示,只有需要任意精度时才付出额外成本。这种"按需付费"的设计,让 1y 既能在脚本场景下足够轻快,又能在科学计算场景下足够精确。

## 为什么是十进制,而不是二进制浮点

1y 选择 `Decimal`(十进制)而非 `binary float`(二进制浮点),是因为**十进制是人类记账的语言**。金融、税务、会计领域的一切数字都是十进制的:`0.1` 美元就是 10 美分,而不是某个无法精确表示的二进制近似。

```1y
# 在 JavaScript 中
# 0.1 + 0.2 === 0.30000000000000004  ← 经典陷阱

# 在 1y 中
let total = 0.1 + 0.2          # Decimal,0.3  ← 正确
let tax = 19.99 * 0.08         # Decimal,1.5992
```

二进制浮点的 `0.1 + 0.2 ≠ 0.3` 是无数金融 bug 的源头。1y 从语言层面消除了这个问题:只要你的字面量是十进制的,运算结果就是精确的十进制。

## 一处心智,通吃所有数值

数值统一的最终收益是**心智负担的消失**。你不再需要:

- 判断该用 `int` 还是 `long` —— 没有 `long`;
- 判断该用 `float` 还是 `double` —— 没有 `float`/`double`;
- 担心大数场景下要不要换 `BigInteger` —— `Int` 本来就是;
- 担心金额要不要换 `BigDecimal` —— `Decimal` 本来就是;
- 担心混合运算的类型转换 —— 自动提升。

剩下你要关心的,只有问题本身。这正是 1y 想要的:语言替你处理机械的细节,让你把注意力留给真正的逻辑。
