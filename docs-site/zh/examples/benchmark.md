# 性能基准测试

使用 `tests/bench_yin.py` 对 yin Web 框架进行并发性能测试。

## 运行基准测试

```bash
# 终端 1：启动服务器
cargo run --release -- run examples/yin_bench_server.1y

# 终端 2：运行压测
python tests/bench_yin.py
```

## 测试结果

测试环境：Windows，release 构建，N-worker 池（N = CPU 核数）。

### 并发性能（GET /ping）

| 请求数 | 顺序 (ms) | 并发 (ms) | 顺序 (req/s) | 并发 (req/s) | 加速比 | 成功率 |
|-------:|----------:|----------:|-------------:|-------------:|-------:|-------:|
| 10     |      64.9 |      33.8 |        154.0 |        296.0 | 1.92x  | 10/10  |
| 100    |     542.7 |     192.0 |        184.3 |        520.9 | 2.83x  | 100/100 |
| 1000   |    4947.6 |     865.4 |        202.1 |       1155.6 | 5.72x  | 1000/1000 |
| 10000  |   44245.3 |    6463.6 |        226.0 |       1547.1 | 6.85x  | 7689/10000 |

### 优化历程

| 版本 | 10000 并发吞吐 | 10000 成功率 |
|------|---------------:|-------------:|
| 原始版（sleep_ms 轮询） | 279 req/s | 1.6% |
| accept_async（mio 事件驱动） | 534 req/s | 99.96% |
| 多线程（N-worker + 批量 accept） | 1547 req/s | 76.9% |

多线程版本在 10000 并发下实现了 **5.5 倍吞吐提升**，1000 并发达到 1156 req/s 且 100% 成功。

### 无色异步验证

发送 1 个慢请求（GET /slow，await 500ms）+ 5 个快请求（GET /ping）并发：所有 5 个快请求在慢请求之前完成，验证了 `await process.sleep_async` 不会阻塞事件循环中的其他处理器。

## 优化技术

### 1. accept_async（mio 事件驱动）

用 `socket.accept_async(listener)` 替代 `process.sleep_ms(1)` 轮询。mio 只在有连接待处理时唤醒调度器，避免空转。

### 2. 多线程 WorkerPool

`WorkerPool` 使用 N 个 worker 线程（N = CPU 核数），每个 worker 预加载入口文件定义后保持存活。通过 `SendValue` 跨线程传递参数和返回值。

### 3. 批量 accept

每次循环 drain 所有待处理连接，减少高并发时每连接的协程开销。

## 未来：字节码 VM

当前 tree-walking 解释器是主要性能瓶颈。字节码 VM 将 AST 编译为扁平指令序列，预计可提升 10-100 倍。这是长期目标。
