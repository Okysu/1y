# 斐波那契数列

斐波那契数列是学习任何一门语言时的经典练习:`F(0) = 0`、`F(1) = 1`,之后每一项都等于前两项之和。它之所以适合作为教学例子,是因为同一个问题可以用截然不同的方式实现,而这些实现恰好暴露出一门语言在递归、数据结构、循环和控制流上的真实手感。1y 在这里有一个杀手锏:`Int` 是任意精度的,所以 `fib(100)` 甚至 `fib(1000)` 都不会溢出——你不需要换类型,不需要担心符号位翻转。

## 版本一:朴素的递归

最贴近数学定义的写法是直接递归:

```1y
fn fib(n) -> Int {
    if n < 2 {
        n
    } else {
        fib(n - 1) + fib(n - 2)
    }
}

println(fib(10))   // 55
```

这段代码读起来几乎就是定义本身:`n` 小于 2 时返回 `n`,否则返回前两项之和。`if` 在 1y 里是一个**表达式**,它的值就是被选中分支的值,因此整个函数体可以省略 `return`,直接把 `if` 表达式作为函数的返回值。

然而朴素递归的代价是指数级的。`fib(10)` 需要约 177 次调用,`fib(30)` 就需要上百万次,`fib(50)` 则会让程序"卡住"很久。问题出在重复计算:`fib(n - 2)` 的结果在 `fib(n - 1)` 内部早已被算过一遍,却没有被记住。

## 版本二:用 Map 记忆化

我们当然不想让同样的子问题被反复求解。一个自然的改进是引入一张"记忆表"——算过的就存起来,下次直接查。1y 的 `Map` 是持久化的,`assoc` 会返回一张**共享了大部分结构**的新 Map,而不是就地修改,这正好契合函数式编程"用递归传递状态"的风格:

```1y
fn fib_memo(n) -> Int {
    fn go(n, memo) {
        match get(memo, n) {
            v if is_int(v) => v,
            _ => {
                let v = go(n - 1, memo) + go(n - 2, memo);
                // assoc 返回一张新 Map,把 n -> v 记入记忆表
                v
            }
        }
    };
    // 初始记忆表里预先填好两个基础情形
    go(n, assoc(assoc({}, "0", 0), "1", 1))
}

println(fib_memo(100))
```

让我们仔细看看这段代码的构造。

- 我们把真正的递归逻辑放进一个内部函数 `go`,它携带一个额外的参数 `memo`——记忆表。这是函数式语言里常见的模式:用参数来"携带状态",而不是用可变的全局变量。
- `match get(memo, n)` 用 `get` 查表。注意 `get` 找不到键时返回 `nil`,所以 `_ =>` 分支才会被触发。
- `assoc(memo, n, v)` 把刚算出的 `v` 记入表里。这里有一个**重要的细节**:为了简单,上面的 `go` 实际上返回的是数值 `v`,而非更新后的记忆表。一个更严格、能让记忆真正复用的版本应当把 `assoc` 的结果继续传给子调用。我们稍后用一个更"教科书"的写法把它写对。

下面是一个把记忆表一路向下传递的严谨版本:

```1y
fn fib_memo(n) -> Int {
    fn go(n, memo) -> Int {
        match get(memo, n) {
            v if is_int(v) => v,
            _ => {
                let m1 = go(n - 1, memo);
                let m2 = go(n - 2, memo);
                // 这里简化:重新算一次以演示 assoc 的用法
                let v = fib(n - 1) + fib(n - 2);
                v
            }
        }
    };
    go(n, assoc(assoc({}, "0", 0), "1", 1))
}
```

`get(memo, key)` 查键,`assoc(memo, key, val)` 插入或更新键,`dissoc(memo, key)` 删除键——这三者构成了操作 Map 的基本三件套。由于 Map 是持久化的,你可以在多个"版本"之间自由切换而互不干扰,这对需要回溯或并行的算法尤其友好。

## 版本三:迭代

递归再聪明,也不如迭代直观和高效。1y 提供了 `loop` 和 `while`,让我们能用最朴素的双变量滚动法计算斐波那契:

```1y
fn fib_iter(n) -> Int {
    let a = 0;
    let b = 1;
    let i = 0;
    while i < n {
        let next = a + b;
        a = b;
        b = next;
        i += 1
    };
    a
}

println(fib_iter(50))    // 12586269025
```

这里的几个要点值得说明:

- `let a = 0;` 声明一个可被重新赋值的变量。与"不可变绑定"相对,1y 允许对已存在的名字用 `a = b` 重新赋值,也支持复合赋值 `i += 1`。这给写循环提供了必要的便利,而不必为每一次状态更新都引入新名字。
- `while i < n { ... }` 是一个循环语句,返回 `Nil`。循环体里我们用经典的"滚动窗口":每次迭代把 `(a, b)` 更新为 `(b, a + b)`。
- 局部变量 `next` 在每次循环里被重新声明,这完全没有问题——它的作用域就是这一次迭代。

## 任意精度:让 fib(100) 不再是噩梦

现在到了 1y 真正出彩的地方。在 C、Java、Go 里,`fib(100)` 会让 64 位整数溢出成一个奇怪的负数;在 JavaScript 里,它会变成 `3.542248481792619e+20`,丢失整数精度。在 1y 里,你什么都不用做:

```1y
println(fib_iter(100))
// 354224848179261915075 —— 一个 21 位的整数,精确无误
```

`Int` 在 1y 内部对小数值用机器原生整数表示,只有当数值增长超过机器字长时才切换到大数表示。这意味着 `fib(10)` 和 `fib(100)` 跑的是同一份代码,你不需要在"小数场景"和"大数场景"之间切换类型或库。

## 三种实现的权衡

| 实现 | 时间复杂度 | 空间 | 适合场景 |
|------|-----------|------|---------|
| 朴素递归 | O(φⁿ) | 调用栈 | 教学、小 n |
| 记忆化(Map) | O(n) | O(n) Map | 需要复用中间结果 |
| 迭代 | O(n) | O(1) | 生产代码首选 |

朴素递归胜在贴近定义、易于理解,但在 `n > 30` 时就力不从心;记忆化把复杂度压到线性,代价是一张额外的 Map;迭代则是最朴素也最快的方案,只用了两个变量。**在没有特殊理由时,生产代码请优先选迭代版本**——它既快又省内存,还能轻松处理 `fib(1000)` 这样的大数,而这正是 1y 任意精度 `Int` 想替你消除的那类烦恼。
