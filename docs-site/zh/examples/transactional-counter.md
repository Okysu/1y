# 事务性计数器

计数器是并发世界里最小的"共享可变状态"问题:多个执行流要对同一个数字加一,如果同步没做好,就会丢更新。线程加锁能解决,却要操心死锁和锁粒度;Actor 能解决,但每次读写都要发消息,且跨多个变量的原子更新不直观。1y 提供了第三条路——**软件事务内存(STM)**。本例用 `shared` + `transact` 构建一个线程安全的计数器,逐步展开快照隔离、冲突重试、嵌套事务,并把它和 Actor 方案做个对比。

## shared:可共享的事务单元

1y 里普通的 `let` 绑定是不可变的。要表达"一块可被并发修改的状态",需要用 `shared` 声明一个事务单元(Transactional cell):

```1y
shared counter = 0;
println(counter);      // 0
```

`shared counter = 0` 创建一个版本化的可变单元,初值为 `0`。读取名字 `counter` 会自动解引用,返回它当前的值;在事务外对它赋值(如 `counter = 10`)会直接写入并提升版本号。但在并发场景下,我们几乎总是把读写放进事务里,以获得原子性。

## 用 transact 做原子自增

最基本的需求是"读—改—写"必须原子:读到旧值、加一、写回新值,这三步要么整体成功,要么整体回滚。`transact` 正是为此而生:

```1y
shared counter = 0;

fn bump() -> Int {
    transact {
        counter = counter + 1;
        counter
    }
}

println(bump());    // 1
println(bump());    // 2
println(bump());    // 3
```

让我们拆解 `bump` 里发生的事:

- `transact { ... }` 开启一个事务。事务体对 `shared` 单元的写入不会立刻生效,而是先缓冲在一个"写集"里。
- `counter = counter + 1` 先读取 `counter` 的当前值(在事务内,这次读取看到的是事务开始时的一致快照),加一,然后把新值记入写集。
- 最后的 `counter` 是事务体的尾表达式,它的值会成为整个 `transact` 的返回值。事务提交时,写集里所有改动一次性原子地写入,并提升相关单元的版本号。

关键在于:**提交时如果发现事务读过的单元已被别的事务改过,整个事务会自动重试**。这就是 STM 的乐观并发——不加锁,冲突了就重跑。对于计数器这种"冲突偶发"的场景,重试的开销几乎可以忽略。

## 快照隔离:读己所写

事务内部能看见自己刚写入的值,这叫"读己所写"(read-your-writes),是快照隔离的一部分:

```1y
shared x = 0;
let snap = transact {
    x = x + 1;       // 写入缓冲:x = 1
    x = x + 1;       // 再次读 x,看到的是缓冲里的 1,加一得 2
    x
};
println(snap);       // 2
println(x);          // 2 —— 提交后对外可见
```

第二次读取 `x` 时,事务返回的是写集里的缓冲值 `1`,而非提交值 `0`。这让事务体内的代码读起来就像单线程顺序执行,你不必在脑子里维护"哪些是已提交的、哪些是缓冲的"——事务替你做好了。

## retry:等到条件成立

`retry` 会放弃当前事务并重新开始,直到它能无冲突地提交。配合"只有 `shared` 写入会被丢弃,`let` 变量跨重试保留"这一规则,可以优雅地实现"等待条件":

```1y
let attempts = 0;
shared ready = 0;

let result = transact {
    attempts = attempts + 1;
    if attempts < 3 { retry };
    ready = 99;
    ready + attempts
};
println(result);          // 102 —— ready(99) + attempts(3)
println(attempts);        // 3 —— let 变量在重试中保留
```

这里 `attempts < 3` 时调用 `retry`,事务从头重跑。由于 `attempts` 是普通 `let` 变量,它不会被回滚,于是每次重试都累加,直到第三次条件不再成立,事务得以提交。这是一个非常优雅的"重试到成功"模式,常用于等待某个共享条件被其他执行流满足。STM 默认最多重试 64 次,超出会抛出异常,避免无限循环。

## 嵌套事务

`transact` 可以嵌套。内层事务的写集会**合并到外层**,而不是独立提交;内层抛出的异常只会回滚内层的写入,外层不受影响:

```1y
shared m = 0;
transact {
    m = m + 1;                          // 外层:m = 1
    try {
        transact {
            m = m + 10;                 // 内层:m = 11
            raise "inner fail"          // 内层回滚
        }
    } rescue as _e {
        nil
    };
    m                                   // 外层仍看到 m = 1
};
println(m);                             // 1
```

内层事务因 `raise` 失败,它的 `m = m + 10` 被丢弃;外层事务继续,最终只提交了 `m = m + 1`。这种"局部回滚 + 全局提交"的语义,让你可以安全地把一段可能失败的逻辑包进内层事务,而不影响外层的整体原子性。

## 与 Actor 方案的对比

同一个计数器,用 Actor 这样写:

```1y
actor Counter {
    state count = 0
    on inc() { count = count + 1; reply(count) }
    on get() { reply(count) }
}

let c = spawn Counter();
c ? inc();
```

两种方案各有侧重:

| 维度 | Actor 计数器 | STM 计数器 |
|------|-------------|-----------|
| 状态封装 | 完全私有,只能靠消息 | `shared` 单元可被多方直接读写 |
| 跨多变量原子更新 | 需要协议,不直观 | `transact` 天然支持,组合性好 |
| 调用方式 | `?` 阻塞请求/回复 | 函数调用,同步返回 |
| 失败处理 | 消息丢失/超时需自行设计 | 自动重试,语义清晰 |
| 适合场景 | 长生命周期有状态服务 | 跨边界的协调式共享访问 |

经验法则和上一例一致:**默认用 Actor,只有当需要原子地协调多个共享引用时才转向 STM**。计数器看似简单,但一旦它要和别的状态一起原子变更(例如"计数 +1 且日志追加一条记录"),STM 的组合性优势就立刻凸显——你只需把两段写操作放进同一个 `transact`,无需重新设计锁的顺序或消息协议。
